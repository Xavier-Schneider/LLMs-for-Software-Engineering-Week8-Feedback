### Problem A: Logging Hides a Race Condition

**Task Description:**
Debug this code. It is supposed to cache a user profile so that concurrent requests only fetch it once, but under concurrency it sometimes fetches the same profile multiple times. Identify the bug and fix it.

**Starter Code:**  
```python
import threading
import time
import random
import statistics

cache = {}
fetch_count = 0

def fetch_profile(user_id: str) -> dict:
    global fetch_count
    fetch_count += 1
    time.sleep(0.001 + random.random() * 0.001)  # fake I/O
    return {"user_id": user_id}

def get_profile_buggy(user_id: str) -> dict:
    # BUG: check-then-act with no lock
    if user_id not in cache:
        cache[user_id] = fetch_profile(user_id)
    return cache[user_id]

def run_trial(threads: int = 20) -> int:
    global cache, fetch_count
    cache = {}
    fetch_count = 0

    barrier = threading.Barrier(threads)

    def worker():
        barrier.wait()
        get_profile_buggy("alice")

    ts = [threading.Thread(target=worker) for _ in range(threads)]
    for t in ts:
        t.start()
    for t in ts:
        t.join()

    return fetch_count

results = [run_trial() for _ in range(20)]
print("Expected correct value: 1")
print("Observed fetch_count values:", results)
print("Average:", statistics.mean(results))
```

```python
import time
import random
import statistics

cache = {}
fetch_count = 0
log_lock = threading.Lock()

        print(msg)
        time.sleep(0.005)  # simulate slow logging / console I/O

def get_profile_buggy(user_id: str) -> dict:
    debug_log(f"{threading.current_thread().name}: checking cache for {user_id}")

    if user_id not in cache:
        debug_log(f"{threading.current_thread().name}: cache miss for {user_id}")
        cache[user_id] = fetch_profile(user_id)

    debug_log(f"{threading.current_thread().name}: returning cached value")
    return cache[user_id]

def run_trial(threads: int = 20) -> int:
    global cache, fetch_count
    cache = {}
    fetch_count = 0

    barrier = threading.Barrier(threads)

    def worker():
        barrier.wait()
        get_profile_buggy("alice")

    ts = [threading.Thread(target=worker, name=f"T{i}") for i in range(threads)]

print("Average:", statistics.mean(results))
```

**Analysis:**
The guideline assumes that adding debug outputs is a harmless way to gain more context. In this program, that assumption is false. The bug is a timing-sensitive race condition, and the added logging changes the thread schedule. Because `debug_log()` is slow and serialized with a lock, threads no longer hit the unsafe cache check at the same time. As a result, the race becomes much less visible, or even seems to disappear.

### 3. New and Updated Guidelines that worked for Problem A

#### Guideline 3: Classify the Bug
**Guideline Definition**
For complex programs, do not automatically add debug outputs first. First classify the bug. If it may be concurrency-sensitive, timing-sensitive, performance-sensitive, or memory-sensitive, avoid naive logging as the first step. Instead, prefer low-perturbation evidence such as minimal repro tests, assertions, counters, stack traces, debugger watchpoints, profilers, or race detectors. Add debug outputs only when they are unlikely to change the program's behavior, and keep them minimal and localized.

**Prompt and Context:**
Context: Above code

Prompt: Debug this code. It is supposed to cache a user profile so that concurrent requests only fetch it once, but under concurrency it sometimes fetches the same profile multiple times. Do not add debug prints first, because the bug may be timing-sensitive. Instead, create a minimal stress test, identify the root cause, and fix it with the smallest correct change. Then show that the fix works.

**Solution or Output generated by GitHub Copilot:**
```
import threading
import time
import random
import statistics

cache = {}
fetch_count = 0

def fetch_profile(user_id: str) -> dict:
    global fetch_count
    fetch_count += 1
    time.sleep(0.001 + random.random() * 0.001)  # simulate I/O
    return {"user_id": user_id}

def get_profile_buggy(user_id: str) -> dict:
    # BUG: unsynchronized check-then-act on shared state
    if user_id not in cache:
        cache[user_id] = fetch_profile(user_id)
    return cache[user_id]

cache_lock = threading.Lock()

def get_profile_fixed(user_id: str) -> dict:
    # Smallest correct fix: protect shared cache access
    with cache_lock:
        if user_id not in cache:
            cache[user_id] = fetch_profile(user_id)
        return cache[user_id]

def run_trial(fn, threads: int = 20) -> int:
    global cache, fetch_count
    cache = {}
    fetch_count = 0

    barrier = threading.Barrier(threads)

    def worker():
        barrier.wait()
        fn("alice")

    ts = [threading.Thread(target=worker) for _ in range(threads)]
    for t in ts:
        t.start()
    for t in ts:
        t.join()

    return fetch_count

def evaluate(fn, trials: int = 30):
    results = [run_trial(fn) for _ in range(trials)]
    return results, statistics.mean(results)

if __name__ == "__main__":
    buggy_results, buggy_avg = evaluate(get_profile_buggy)
    fixed_results, fixed_avg = evaluate(get_profile_fixed)

    print("Expected correct value per trial: 1")
    print()
    print("Buggy version:")
    print("  sample results:", buggy_results[:10])
    print("  average fetch_count:", buggy_avg)
    print()
    print("Fixed version:")
    print("  sample results:", fixed_results[:10])
    print("  average fetch_count:", fixed_avg)
```
**Analysis:**
The new guideline worked because it avoided adding logging to a timing-sensitive bug. Instead, it used a stress test to reproduce the issue, identified the unsynchronized access to cache, and fixed it by adding a lock. The same test then showed the fix worked: the buggy version fetched multiple times, while the fixed version fetched only once.

### 3. New and Updated Guidelines that worked for Problem B

#### Guideline #7: Instruct the LLM to write tests and keep iterating on a fix until the code passes the tests
**Guideline Definition**  
Instruct the LLM to write tests and keep iterating on a fix so that it won't just implement a fix and confidently move on thinking it solved the problem. The tests give it feedback on whether it actually solved the problem. If it did not, it should keep iterating until it comes up with a workable solution.

**Prompt and Context:**  
Context:  
The code in the ProblemB folder.

Prompt:  
I have an algebra solver written in Python. It is currently failing to find a solution with the input 'solve 3x = 0'. Write tests for this case and keep iterating on a fix until the code passes the tests.

**Solution or Output generated by GitHub Copilot:**  
In this case GPT-4.1/Copilot first sets up a test file called 'test_solver.py':
```
import unittest
from solver import solve

class TestSolver(unittest.TestCase):
    def test_solve_3x_eq_0(self):
        # Expect x = 0 as the solution
        from solver import solve_equation
        result = solve_equation('3x = 0')
        self.assertIn('0', result['solutions'])

if __name__ == '__main__':
    unittest.main()
```

It then proceeds to call the test file by running the command:
```
python3 ProblemB/test_solver.py
```

The test file is able to run flawlessly as it imports the 'solve_equation' function directly and executes the function rather than trying to execute the main program. The test originally failed, so the agent went looking for the solution. It then decided to change line 60 in the solver.py file to:

```
solutions = [s for s in solve(equation, var) if s is not None]
```

Once the agent made that change, the tests passed and it was happy to report back that it had fixed the problem.

**Analysis:**  
By telling the LLM to create tests and iterate on the code until the tests pass, the agent is less likely to get stuck in scenarios where the main program requires user input from the command line. Tests also give the agent instant feedback on whether or not its changes actually had a positive impact on fixing the given problem, rather than the agent assuming it fixed the problem without actually doing so and moving on to the next task.

Note: the code for the Algebra solver was written by Opus 4.6.

### Problem C: Token Refresh Race + Expiry Drift

**Task Description:**  
Debug this code. It implements an API client that caches an access token and refreshes it when expired. Under concurrency, it sometimes:
- Refreshes the token multiple times unnecessarily.
- Uses an expired token briefly.
- Rarely raises a KeyError or returns None.

Identify the bug(s) and fix them safely.

**Starter Code:**  
```python
import threading
import time
import random
from typing import Optional

# Simulated token server
refresh_count = 0

def fetch_new_token() -> dict:
    global refresh_count
    refresh_count += 1
    time.sleep(0.002 + random.random() * 0.002)  # simulate latency
    return {
        "access_token": f"TOKEN-{refresh_count}",
        "expires_at": time.time() + 0.01  # very short TTL
    }

class APIClient:
    def __init__(self):
        self._token: Optional[dict] = None

    def _is_expired(self, token: dict) -> bool:
        return time.time() >= token["expires_at"]

    def get_token_buggy(self) -> str:
        """
        Return a valid access token.
        Should:
          - Refresh only once when expired
          - Never return expired token
          - Be thread-safe
        """

        # Case 1: No token yet
        if self._token is None:
            self._token = fetch_new_token()
            return self._token["access_token"]

        # Case 2: Expired
        if self._is_expired(self._token):
            self._token = fetch_new_token()

        return self._token["access_token"]


# ---- Concurrency test ----

def run_trial(thread_count=30):
    global refresh_count
    refresh_count = 0

    client = APIClient()
    barrier = threading.Barrier(thread_count)

    def worker():
        barrier.wait()
        for _ in range(5):
            client.get_token_buggy()
            time.sleep(random.random() * 0.001)

    threads = [threading.Thread(target=worker) for _ in range(thread_count)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    return refresh_count

if __name__ == "__main__":
    trials = [run_trial() for _ in range(10)]
    print("Expected refresh count: close to minimal")
    print("Observed refresh counts:", trials)
```

### 3. New and Updated Guidelines that worked for Problem C

#### Guideline 6: Constraint Anchoring and Explicit Edge Case Verification
**Guideline Definition**
When prompting the LLM, explicitly enumerate all critical constraints and edge cases at the beginning of the prompt.  Require the model to explicitly verify its solution against each constraint.  Reinforce the most important constraints again at the end of the prompt to prevent omission.  Avoid vague phrases like “keep edge cases in mind” without listing what those edge cases are. The goal is to anchor reasoning around clearly defined requirements rather than relying on recency bias alone.

**Prompt and Context:**  
```
Debug the code. The token refresh logic is incorrect under concurrency.

Fix the implementation.

## Must satisfy ALL constraints

1. Never return an expired token.
2. At most one refresh may run concurrently.
3. Do not hold a lock during the network call.
4. Handle expiry during the refresh window.
5. Must work under very short TTL (5–10 ms).
6. No deadlocks.
7. No major performance degradation.
8. Do not modify unrelated parts of the file.

## Required output

1. Brief root cause analysis.
2. Minimal code changes only.
3. Explicitly verify the fix against each constraint above.

Keep edge cases in mind.
The solution must satisfy all constraints simultaneously.
```

**Solution or Output generated by GitHub Copilot:**  

```python
import threading
import time
import random
from typing import Optional

# Simulated token server
refresh_count = 0

def fetch_new_token() -> dict:
    global refresh_count
    refresh_count += 1
    time.sleep(0.002 + random.random() * 0.002)  # simulate latency
    return {
        "access_token": f"TOKEN-{refresh_count}",
        "expires_at": time.time() + 0.01  # very short TTL
    }

class APIClient:
    def __init__(self):
        self._token: Optional[dict] = None
        self._lock = threading.Lock()
        self._refreshing = False
        self._cond = threading.Condition(self._lock)

    def _is_expired(self, token: dict) -> bool:
        return time.time() >= token["expires_at"]

    def get_token(self) -> str:
        """
        Return a valid access token (thread-safe, correct refresh logic).
        """
        while True:
            with self._lock:
                token = self._token
                if token is not None and not self._is_expired(token):
                    return token["access_token"]
                if not self._refreshing:
                    self._refreshing = True
                    break
                # Wait for refresh to complete
                self._cond.wait()
            # Only one thread gets here: do refresh without holding lock
            new_token = fetch_new_token()
            with self._lock:
                self._token = new_token
                self._refreshing = False
                self._cond.notify_all()
                # Double-check expiry in case TTL is extremely short
                if not self._is_expired(self._token):
                    return self._token["access_token"]
        # If expired, loop and try again

    def get_token_buggy(self) -> str:
        """
        Return a valid access token.
        Should:
          - Refresh only once when expired
          - Never return expired token
          - Be thread-safe
        """

        # Case 1: No token yet
        if self._token is None:
            self._token = fetch_new_token()
            return self._token["access_token"]

        # Case 2: Expired
        if self._is_expired(self._token):
            self._token = fetch_new_token()

        return self._token["access_token"]


# ---- Concurrency test ----

def run_trial(thread_count=30):
    global refresh_count
    refresh_count = 0

    client = APIClient()
    barrier = threading.Barrier(thread_count)

    def worker():
        barrier.wait()
        for _ in range(5):
            client.get_token()  # Use fixed method
            time.sleep(random.random() * 0.001)

    threads = [threading.Thread(target=worker) for _ in range(thread_count)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    return refresh_count

if __name__ == "__main__":
    trials = [run_trial() for _ in range(10)]
    print("Expected refresh count: close to minimal")
    print("Observed refresh counts:", trials)
```

```
Root cause analysis:

The current implementation of get_token_buggy is not thread-safe. Multiple threads can observe an expired token (or no token), then all proceed to refresh in parallel, causing:

Multiple unnecessary refreshes (violates constraint 2).
A race where a thread may read and return an expired token if another thread is refreshing (violates constraint 1).
No lock is used, so there is no concurrency control.
If a lock is naively added and held during the network call, it will degrade performance and risk deadlocks (violates constraints 3, 6, 7).
Next steps:

Design a minimal fix using a lock and a condition variable to ensure only one refresh runs at a time, no expired token is returned, and the lock is not held during the network call.
Implement the fix in a new method (to avoid modifying unrelated code).
Explicitly verify the fix against all constraints.
Proceeding to design and implement the fix.

Root cause analysis (summary):

The original code allowed multiple threads to refresh the token simultaneously and could return expired tokens due to race conditions. No lock was used, so concurrency was not controlled.
Minimal code changes made:

Added a lock and condition variable to ensure only one refresh runs at a time.
No lock is held during the network call.
All threads wait for a valid token if a refresh is in progress.
Double-checks expiry after refresh to handle very short TTLs.

Verification against constraints:

Never return an expired token:
The method always checks expiry before returning, including after refresh.

At most one refresh may run concurrently:
Only one thread sets _refreshing = True and performs the refresh; others wait.

Do not hold a lock during the network call:
The lock is released before calling fetch_new_token().

Handle expiry during the refresh window:
After refresh, expiry is checked again; if expired, the process repeats.

Must work under very short TTL (5–10 ms):
Handles expiry checks before and after refresh, and loops if needed.

No deadlocks:
Condition variable is always notified, and the lock is not held during network calls.

No major performance degradation:
Lock is only held for short critical sections; no lock during network I/O.

Do not modify unrelated parts of the file:
Only the APIClient class and the test invocation were minimally changed.

Edge cases:

Handles token expiry during/after refresh, and high concurrency with very short TTLs.
The implementation now satisfies all constraints.


```

**Analysis:**

The improved guideline clearly listed all constraints and required the model to check each one. Because of this, the model had to think more carefully about concurrency, expiry timing, and performance. It used a condition variable and controlled the refresh process properly instead of just adding a simple lock. It also verified the solution against each requirement step by step. This shows that clearly listing constraints and forcing verification leads to better and more reliable solutions.
